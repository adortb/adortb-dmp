### 2.2 内置标签规则（10 条）

规则在 `internal/tagging/rule.go` 中定义，引擎在 `internal/tagging/engine.go` 中并发驱动。

| 规则编号 | 规则分类 | 触发依据（示意） |
|----------|----------|----------------|
| 1 | 兴趣标签 | 特定类目点击/浏览频次 |
| 2 | 高价值用户 | 消费金额/转化次数 |
| 3 | 购买意向 | 加购/结算行为 |
| 4 | 活跃用户 | 近期活跃天数 |
| 5 | 设备特征 | 设备类型/系统版本 |
| 6-10 | 其他维度 | 详见 `tagging/rule.go` |

### 交互协议说明

| 依赖服务 | 协议 | 连接方式 | 说明 |
|----------|------|----------|------|
| Redis | RESP over TCP | 连接池（50） | 标签读写、事件计数 |
| ClickHouse | HTTP API | http.Client | 批量事件归档（100条/5s） |
| Kafka | Wire Protocol over TCP | 长连接 | 多 topic 并发消费，5s 重连 |
| DSP | HTTP/JSON | client 库 | 批量标签查询，P99<10ms |
| CDP | Kafka | 消息生产 | 发布 `adortb.dmp.tags` topic |

## 4. 批量标签查询时序图

场景：DSP 在竞价前通过 `POST /v1/tags/batch` 批量查询最多 200 个用户的标签。

```
DSP (client.go)          api/server.go        lookup/service.go       Redis
      │                        │                       │                 │
      │── POST /v1/tags/batch ─▶│                       │                 │
      │   Body: {user_ids:[...]}│                       │                 │
      │                        │                       │                 │
      │                        │── BatchTags(ctx,[]) ──▶│                 │
      │                        │                       │                 │
      │                        │                       │─ 并行 HMGET ────▶│
      │                        │                       │  (Pipeline,      │
      │                        │                       │   批量 key)      │
      │                        │                       │                 │
      │                        │                       │◀── 批量结果 ─────│
      │                        │                       │                 │
      │                        │                       │── 合并/反序列化 ─│
      │                        │                       │   (goroutine 分片│
      │                        │                       │    并行处理)     │
      │                        │                       │                 │
      │                        │◀── map[uid][]Tag ─────│                 │
      │                        │                       │                 │
      │◀── 200 OK JSON ─────────│                       │                 │
      │    {results:{...}}      │                       │                 │
      │                        │                       │                 │

关键性能保障：
  1. Redis Pipeline 批量发送，减少 RTT
  2. 用户列表分片，多 goroutine 并行查询
  3. 结果合并无锁（各 goroutine 写独立 slice，主 goroutine 汇总）
  4. P99 目标：< 10ms（≤200 用户）
```

### ClickHouse 写入流程

```
行为事件
   │
   ▼
store/clickhouse.go
   │
   ├── 写入内存 buffer（chan，容量 100）
   │
   ├── 触发条件（满足任一）：
   │   ├── buffer 达到 100 条
   │   └── 距上次 flush 超过 5 秒
   │
   └── HTTP POST 批量写入
       http://clickhouse:8123/?query=INSERT+INTO+...
```

## 7. 受众扩展算法

```
输入：种子标签列表 [tag_A, tag_B, tag_C]

Step 1: 从 Redis 并行获取各标签的用户集合
  ZRANGEBYSCORE tag:users:tag_A -inf +inf
  ZRANGEBYSCORE tag:users:tag_B -inf +inf
  ZRANGEBYSCORE tag:users:tag_C -inf +inf

Step 2: 统计每个用户命中的标签数
  user_001: 命中 tag_A, tag_C → 2
  user_002: 命中 tag_A, tag_B, tag_C → 3
  user_003: 命中 tag_B → 1

Step 3: 按命中数降序排列
  [user_002(3), user_001(2), user_003(1)]

Step 4: 排除种子用户（可选），返回扩展用户列表

输出：按相似度排序的扩展用户列表
```
